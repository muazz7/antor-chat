Antor Architecture & Ops

System Documentation & Online Deployment Guide

A senior architectural handbook detailing the design, tech stack, real-time message flow, and production scaling strategy for the Antor Private Chat platform.

System Architecture & Tech Stack

Antor is engineered as a secure, fast, event-driven web application. The core architectural philosophy leverages asynchronous I/O and light database operations to scale elegantly on minimal server resources.

  • Backend Engine: Node.js providing the asynchronous, non-blocking single-threaded event loop environment.
  • Web Application Framework: Express.js managing API endpoints, serving the compiled static assets, and housing standard session management protocols.
  • Database Engine: SQLite run natively via better-sqlite3. It is optimized under Write-Ahead Logging (WAL) mode to enable high-concurrency read operations while processing active write streams.
  • Cryptographic Layers: bcryptjs used to apply hardware-resilient key-derivation hashing to room passwords.
  • Client Layer: Optimized HTML5 layout featuring native visual viewport listening algorithms that gracefully handle mobile keyboard events on iOS & Android devices.

Real-Time Communication Flow

Real-time interactions are powered by Socket.io using persistent HTTP/1.1 WebSocket connections. Here is the operational message and status cycle:

sequenceDiagram participant C as Client Browser participant S as Express & Socket.io Server participant DB as SQLite Engine C->>S: HTTP Request /chat/:slug S-->>C: Serve compiled markup & assets C->>S: Request Connection Upgrade (ws://) S-->>C: Socket Handshake Approved C->>S: emit("join-room", slug) Note over S: Server signs socket to virtual channel rect rgb(24, 28, 36) Note over C, S: Message Broadcast Flow C->>S: emit("send-message", { content }) S->>DB: INSERT INTO messages (sqlite write) S-->>C: broadcast("new-message") to room channels end

Note on Room Isolation: Sockets are cataloged inside isolated logical constructs using Socket.io's built-in socket.join(roomSlug). Broadcasts are strict and scoped utilizing io.to(slug).emit(...) to guarantee zero cross-room data leaks.

Taking the Application Online

Running the server locally differs from orchestrating a secure production environment. To deploy the system online securely and at scale, the following system-level changes are recommended:

1. Horizontal Scaling & Process Distribution

Node's single-threaded nature means it runs on a single CPU core. In production, we run multiple server processes using a process manager or cluster. However, sockets are in-memory. If Client A is on Server Process 1, and Client B is on Server Process 2, they cannot communicate directly.

  • Resolution: Attach a Redis Adapter (@socket.io/redis-adapter). Sockets will transparently distribute messaging events across all worker processes using Redis Pub/Sub mechanics.

2. Production Database Integration

SQLite stores data in a local file. On ephemeral virtual machines (such as AWS Fargate, GCP Cloud Run, or Kubernetes pods), the filesystem is wiped on restart, destroying data.

  • Resolution: Transition the persistence layer from SQLite to PostgreSQL or a managed relational database server. Use connection pool configurations to maintain stable database connections.

3. Reverse Proxy & SSL Offloading (Nginx)

It is bad practice to expose the Node.js port directly to the internet. We install Nginx or Cloudflare in front of the application to serve as a reverse proxy, rate-limiter, and SSL handshaker.

Nginx WebSocket Upgrade Configuration:

server {
    listen 80;
    server_name chat.yourdomain.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        
        # Upgrade headers to support persistent WebSocket streams
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        
        proxy_cache_bypass $http_upgrade;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
